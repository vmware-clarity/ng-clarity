AppFx Datagrid is a data-bound list control that displays items from an array-based source in the form of columns and rows. The datagrid control allows users to select, sort, and filter to quickly find or report on their business needs. Dataset sizes can scale upward without impacting the UI performance.

Key Benefits

  • Reduced Markup: Requires less markup compared to vanilla clr-datagrid, simplifying the development process.
  • Consistent UX: Provides a common look and feel across grids.
  • Centralized Concerns: Handles filtering, sorting, pagination, row actions, A11y, L10n, and zoom support centrally.
  • Advanced Features: Includes advanced filters, export, and selection persistence.
  • Easier Migration: Simplifies migration for applications built on Clarity.

Advanced Features

The component is built on top of the clr-datagrid component from Clarity and provides additional advanced features.

Column Reordering

The datagrid supports column reordering out of the box using the Angular CDK.

Row Drag-and-drop

Row drag-and-drop functionality can be enabled supporting multiple items.

Action Bar

The datagrid has a customizable responsive action bar component: appfx-datagrid-action-bar.

Datagrid Filters

AppFx Datagrid Filters component provides an enhanced user experience for datagrid filtering. It is expected to be hosted in the datagrid action bar and supports three modes: quick, advanced, and advanced only.

Usage

Advanced modes allow users to build and manage complex filtering queries, while Quick mode is suitable for fast, text-based filtering.

Client Side

The Client Side Datagrid example demonstrates how to implement a datagrid where all data is loaded into the client application at once. In this mode, the datagrid component handles pagination internally, while filtering and sorting can be managed via simple event bindings on the client.

Key Features

  • Client-Side Data Binding: The full dataset is passed directly to the [gridItems] input. The grid automatically manages page slicing[cite: 1313].
  • Configuration Models: Layout and footer visuals are controlled via structured input objects (layoutModel, footerModel) rather than multiple individual flags.
  • Client-Side Export: The clientSideExportConfig enables the built-in CSV export functionality for the loaded data.
  • Custom Filtering: The (searchTermChange) event allows you to implement custom filtering logic in your component to update the displayed items.

Usage Configuration

To configure a client-side grid, ensure serverDrivenDatagrid is set to false (this is the default behavior). You can then utilize the GridLayoutModel and GridFooterModel interfaces to customize the grid's appearance.

Server Driven

The Server Driven Datagrid example demonstrates how to handle large datasets where pagination, filtering, and sorting are performed on the server. In this mode, the component acts as a view, emitting events when the user interacts with the grid (e.g., changing the page), which the application must handle to fetch the correct slice of data.

Core Concepts

  • Server-Driven Mode: Enabled by setting [serverDrivenDatagrid]="true". This tells the grid not to slice the provided array but to display it as-is, assuming it represents the current page.
  • Data Fetching: The (refreshGridData) output emits a ClrDatagridStateInterface object whenever the grid state changes (page, sort, filter). You must subscribe to this event to fetch the corresponding data from your backend.
  • Total Items: You must provide the [totalItems] input so the paginator can calculate the total number of pages.
  • Loading State: Bind the [loading] input to show a spinner while data is being fetched.
  • Preserving Selection: When data is replaced (e.g., on page change), selection is normally lost. By using the appfxPreserveSelection directive (or similar) and defining [trackByGridItemProperty], the grid can retain selections across data refreshes.
Example

Virtual Scroll

The Virtual Scroll Datagrid example demonstrates how to efficiently render large datasets by only creating DOM elements for the rows currently visible in the viewport. This example combines virtual scrolling with server-driven data fetching.

Core Concepts

  • Virtual Scrolling Mode: Enabled by setting [virtualScrolling]="true". This requires the grid to also be server-driven ([serverDrivenDatagrid]="true").
  • Data Range Binding: Unlike standard grids, virtual grids use the [dataRange] input instead of [gridItems]. This object contains the current slice of data, the total count, and the skip index.
  • Data Fetching: The (refreshVirtualGridData) output is emitted when the user scrolls to a new position that requires data fetching. It provides the new page range (start/end or skip/size).
  • Layout Requirements: Virtual scrolling works best with a fixed height container. Set stretchToParentHeight: true in the layout model to ensure the grid fills its container.
Example

Column Definitions

The Dynamic Column Definitions example demonstrates how to programmatically add and configure columns in the datagrid at runtime. This is useful for scenarios where the grid structure depends on user input or external data sources.

Key Features

  • Dynamic Column Creation: Columns can be added to the [columns] input array at any time. The grid automatically updates to reflect the new structure.
  • Runtime Configuration: You can configure properties like displayName, width, hidden, and hideable dynamically before adding a column.
  • Conditional Features: Sorting (sortComparator) and filtering (stringFilter) can be conditionally enabled for each new column.
  • Data Binding Updates: When adding a new column that requires new data fields, you must also update the gridItems array to include the corresponding property values for each row.
Example

Detail Pane

The Detail Pane provides a dedicated space within the datagrid to display additional information or actions related to a specific row. This is useful for master-detail patterns where you want to keep the user in the same context.

Key Features

  • Template Injection: Content for the detail pane is defined using standard Angular <ng-template> elements and passed to the grid via the [detailHeader] and [detailBody] inputs.
  • Row Context: The templates receive the row data context (accessed via let-context), allowing you to display dynamic data specific to the active row.
  • State Control: The visibility of the detail pane is controlled by the detailState input, which holds the data item of the currently expanded row (or null if closed).
Example

Row Drag-Drop

The Drag and Drop example demonstrates how to enable dragging rows from the AppFx Datagrid into an external container. This functionality leverages the Angular CDK Drag and Drop primitives and coordinates them using a shared grouping service.

Key Features

  • Datagrid Configuration: The [dragConfig] input enables drag support on the grid. It specifies which data field to display in the drag ghost element.
  • Group Coordination: The DragAndDropGroupService is used to link the datagrid's internal drag source with external cdkDropList containers by registering them to the same unique group string.
  • Multi-Selection Drag: The component supports dragging multiple selected rows simultaneously.

Filters

The Filters example demonstrates how to integrate both Quick and Advanced filtering modes into the AppFx Datagrid. It shows how to define filterable properties, switch between filter modes, and handle filter change events to update the displayed data.

Key Features

  • Filter Modes: Supports switching between Quick (search bar), Advanced (composite filters), and AdvancedOnly modes using the [filterMode] input.
  • Filterable Properties: Defines which columns are available for advanced filtering using FilterablePropertyDefinition objects (e.g., String, Enum).
  • Event Handling: Listens to (searchTermChange) for quick search updates and (advancedFilterChange) for complex query updates.
  • Custom Logic: Demonstrates how to manually filter the dataset based on the emitted criteria and operators (Equals, Contains, etc.).

Persistance

The Persistence feature allows the datagrid to save its state (column visibility, width, order, sorting, and page size) to a storage mechanism (e.g., local storage). This ensures that the user's preferences are retained across sessions or page reloads.

Key Features

  • State Preservation: Automatically saves and restores column configuration (width, visibility, order), sort order, and page size.
  • Unique Identification: Uses a unique key passed to the [appfxPersistDatagridSettings] directive to distinguish settings between different grids.
  • Pluggable Storage: The storage mechanism is agnostic and must be provided via the appfxDatagridPersistSettingsToken. This allows you to implement any storage strategy (Local Storage, Session Storage, Backend API, etc.).
To enable persistence, you must provide a service implementing PersistDatagridSettingsService via the appfxDatagridPersistSettingsToken in your module or component providers. Example

FilterablePropertyDefinition

Base abstract class for filterable properties.

Property Type Description
displayName string Name as shown to users.
property string Actual name of the property that is used for filtering.
supportedOperators ComparisonOperator[] Optional. List of comparison operators. overrides defaults.

StringPropertyDefinition

Extends FilterablePropertyDefinition.

Property Type Description
singleCondition boolean Optional. Indicates that the property can be used with single condition only. Default: false.
logicalOperator LogicalOperator Optional. Restricts to only AND or OR. Default: undefined (supports both).

Default operators: Equals, DoesNotEqual, Contains, DoesNotContain, StartsWith, EndsWith, IsEmpty.

EnumPropertyDefinition

Extends FilterablePropertyDefinition.

Property Type Description
values Map<string, string> Enumeration key/value data to be used in the filter selection.
singleSelect boolean Optional. Indicates single select filtering. Default: false.

NumericPropertyDefinition

Extends FilterablePropertyDefinition.

Property Type Description
singleCondition boolean Optional. Indicates that the property can be used with single condition only. Default: false.
logicalOperator LogicalOperator Optional. Restricts to only AND or OR. Default: undefined (supports both).
unit Unit Default unit of the property metric. Optional.

Default operators: Equals, DoesNotEqual, LessThan, LessThanOrEqualTo, GreaterThan, GreaterThanOrEqualTo, IsEmpty.

DateTimePropertyDefinition

Extends FilterablePropertyDefinition.

Default operators: Equals, Before, BeforeOrEqualTo, After, AfterOrEqualTo, CustomRange, IsEmpty, LastDay, LastWeek, LastMonth, LastYear, TimeSpan.
