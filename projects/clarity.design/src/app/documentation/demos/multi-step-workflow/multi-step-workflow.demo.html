A multi-step workflow is a process in which there are many individual parts, typically in a series. Scenarios include creating an object, setting up a configuration, or onboarding to a new service.

Usage

Use a multi-step workflow when the users want to accomplish a goal with many steps or need complete steps in a specific sequence. It also helps guide novice users or those needing to complete an infrequent process. By showing less information at a time, multi-step workflows allow users to focus better on the content pertinent to each step and decrease the chance of errors.

For more straightforward tasks or when the audience is advanced and doesn’t need guidance, multi-step workflows can slow them down. Consider using an alternative, like a form. Multi-step workflows are also impractical if you try to teach a process or educate the user with text-heavy content because the user is not consuming any additional knowledge when going through a workflow. Instead, their primary focus is to finish the workflow. If education is your goal, try creating an informative tutorial instead.

Here is a quick comparison between when to use a multi-step workflow vs. a form.

Use multi-step workflows

  • For complex workflows with 2+ steps
  • For novice users or infrequent processes
  • When there are dependencies between steps
  • When it needs to be completed in a specific order
  • When the user may leave and come back to complete

Use forms

  • For non-complex workflows
  • For advanced users
  • If the user need to view everything in one page or refer to the content back and forth

Types of multi-step workflows

Basic workflows

Stepper and Timeline

Both the stepper and timeline components are in-page, multi-step patterns that allow users to navigate freely to other parts of an application during the workflow. Both make it easy to resume if users quit the process midway and provide a less constrained workspace when compared to wizards. Because of the bigger workspace, steppers and timelines can also display more complex content in each step.

An application screen with compact and standard line height.

Steppers are good for many steps with small data set in each step. Steppers provide additional context of each steps, but it also limit the vertical space for the dataset within.

Timelines offer more space for longer forms shown on a fewer number of steps due to horizontal constraints.

A timeline can be used in either a full page or small space (e.g. nested inside an expandable datagrid).

Wizard

A wizard helps the user stay focused on the task by presenting the workflow in a separate modal. Only use a wizard when the content is self-sufficient since it blocks access to other parts of the app that are necessary for completing the process. The user will be brought back to the original context after the workflow is finished. This means the entry point and exit point of a wizard are always the same.

Use wizards for workflows with 3-10 steps. They come in three different sizes to fit different screen sizes and content.

Be mindful when using wizards since they are not gracefully disruptive and introduce more cognitive load. If users quit the process midway, they might lose their work and may need to click through the preceding steps again. Don't include long or complex content in each step to avoid scrolling and truncation

Nested workflows

A nested workflow is a long and complex workflow that can be broken down into several smaller workflows. Some workflows may involve multiple products or parties (other users or the system) to complete. It is a good idea to include an overview page to set the expectation of the entire process. The overview page shows the nested workflow's progress with each step's status, actions, and descriptions. Consider showing the user any prerequisites beforehand to avoid the need to leave midway to find additional information. It is important to show progress, considering if the user leaves the workflow and forgets where they were. If the user cannot finish the entire nested workflow at once, provide ways to save their progress (or even better, save it automatically and indicate to them that their progress has been saved) and let the user resume the workflow when ready.

Possible implementations of nested workflow include timeline + steppers, overview page + steppers, etc.

Timeline + stepper

The user can view the overall progress all the time. Each step in the timeline consisted of a basic workflow.

Overview page + stepper

  • The overview page give the user an high level expectation
  • Each step is consisted of a basic workflow

After the basic workflow is completed, it returns to the overview page for the next step

Building a multi-step workflow

Breaking down content

After gathering the required information for the multi-step workflow, the first step is to consolidate all the information. Decide what the required tasks to achieve the user goals versus what are the optional ones are. Eliminating unnecessary optional tasks reduces user effort and increased completion rates. The user can always perform the optional tasks after completing the workflow.

When building a multi-step workflow, break up the task into a series of chunks, or groups of operations. Unrelated tasks should be separated into a new step. This helps preserve context around the step being completed. Also, balance the amount of the content and the number of steps.

Ordering steps

Linear vs. non-linear

Linear workflow requires the user to complete the entire workflow in a prescribed order. It works best when they are dependencies between steps or the steps fall into a logical sequence. Wizard, stepper, and timeline can all support the linear use case.

Use a non-linear workflow if there are no dependencies between steps or the user should be able to choose their path. The order of the tasks “suggests” the order to do the steps, without locking the user into that order (if you have to restrict the order, dynamically disable the controls that launch the steps as necessary).

Example of a non-linear workflow. The user can start either step 3 or 4.

Conditional workflows

Conditional workflows change step content or the number of steps based on the entry from previous steps. Non-conditional wizards are preferable, but if it’s not an option, use a conditional wizard and update the steps following the input. This will cue the user to the changes.

If the selection of the first step changes the following steps significantly, consider separating the first step in a stand-alone page to avoid content shifting.

Selecting the component

Select the appropriate component depending on your use cases.

Moving between steps

Show progress. Because of the complexity of the multi-step workflow, it is important to clearly indicate the progress at all times.

Provide incremental validation. Incremental validation can help the user discover and recover mistakes early. Common types of input, such as email addresses and dates, can be validated when the user tab or click away from the component. Otherwise, a validation after each step can also help the user recover while still in context and prevent errors in later steps with dependencies.

Validate email format when users tab away

Run server-side validation when users click NEXT

Let users return to previous steps. The user should be able to go back to previous steps to view or modify the content. If changes to the previous step have dependencies on later steps, the user will need to input it again.

Allow users to dismiss the workflow.

A user should be able to actively dismiss a workflow. Completing all of the steps or canceling should dismiss the wizard.

Provide a way to save progress.

If the user needs to leave the workflow halfway, it is ideal to auto-save or provide a manual way to save the state so that the user can resume at a later time. However, if the progress cannot be saved because of technical reasons, make sure to warn the user of the consequences of losing the data.

Submit a workflow

Review before complete. It is good practice to present a review of choices made throughout the workflow near the end. This will allow the user to review and double-check the input data before the final button is clicked. If users need to change the data entered, they should be able to navigate back to the given page where the date was entered. If the amount of steps in the wizard is greater than 8, it is a good idea to provide links directly to the screen of the data input.

Validation

The entire workflow may be validated once it has been completed, or on submitted. This is appropriate for secure and server side validation cases. Complex validation such as cases where objects are interdependent, and in cases where to data must be handled securely are best handled on the server side.

When an error occurs, bring the user back to workflow, if it can be corrected. Also, the validation can be done within a few seconds. Otherwise, show a clear error message to the user and provide clear instructions on how to resolve it.
