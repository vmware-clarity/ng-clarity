This pattern documentation covers usage and use cases for notification needs. The components shown are not yet available in the Clarity libraries but have been validated by implementation in VMware applications.

A notification is the system provides a range of methods for communicating with the user. Notifications let the user know when something needs attention by alerting the user to events that have occurred, and by providing guidance about the actions that need to be taken.

To ensure that users don't become frustrated, keep the following guidelines in mind:

  • Ensure that notifications are relevant and timely.
  • Use the right notification for the right situation.
  • Ensure that the component or UI element used to notify users matches the event's priority.

Notification Channels

Notifications can be delivered in a variety of channels: email, text message, in-application, web hooks, etc. When delivering notifications offline (when users aren't in-app), be sensitive to deliver only those notifications which complement the users' sense of urgency. For example, sending a promotional offer via text message would violate the user's sense of urgency in relation to that channel.

In-App Notifications

How and when to deliver Notifications is based on their priority to the User.

Snackbars

Snackbars are quick conformations of successful user actions. (e.g. Form submission, edit object successful, delete object successful) snackbar details

Alerts

Alerts communicate low-priority events related to specific objects or parts of an app. (e.g. Non-disruptive maintenance, configuration suggestions) alert details

Toasts

Toasts communicate medium-priority events that the user may want to reference later in the notifications panel. (e.g. Approval needed, sddc created) toast details

Banners

Banners are critical notifications that require user action but do not prevent the user from continuing to use the app. (e.g. Payment problem) banner details

Modals

Modals are the highest priority notification: since they disrupt user task flows, they should be reserved for critical events that require user action (e.g. Account Suspended) modal details

Snackbar

Snackbars are quick confirmations of successful user actions.

Snackbars are low in the visual/interaction hierarchy because users are aware of the request they have submitted, and can visit the associated object later elsewhere in the UI (ex. after adding a user, I can see that the user has been added in the "Users" list).

Anatomy

  1. Icon: Success Icon. For errors, warnings, or information, use Toast or Banner
  2. Description (max: 156 characters): A description of the confirmation event.
  3. Link (max: 25 characters): When possible, provide a link to details describing the event.
  4. Dismiss: An icon allowing the user to remove the toast from their view. Do not use "dismiss" as a link, clicking the dismiss button dismisses the snackbar.

Interaction

Usage Guidelines

  • Use snackbars to communicate successful actions.
  • Provide a link to the created/modified/updated object when possible.
  • Allow the user to dismiss the snackbar.
  • Remove the snackbar after a 6 second timeout if the user has not already dismissed it. This allows the snackbar to provide an ephemeral confirmation that does not disrupt from the user's ongoing tasks.
  • Remove the snackbar if the user navigates away from the current view.
  • When an action has updated multiple records, communicate the number of affected records in the messaging when the records are greater than 2 ("johndoe@gmail.com, annedoe@gmail.com, and 5 other users added.")
  • For accessibility considerations, allow the user to set the length of time disappearing objects remain in the viewport. Additionally, allow the user to turn off UI motion overall.
  • Use "View Details" as the link title in most cases.
  • When the snackbar confirms multiple elements, only show a link if the destination that references all elements.
  • Leave snackbars on screen, unless the user has explicitly configured that as an accessibility setting.
  • Use more than a single link in the snackbar. The snackbar provides a single link for accessbility purposes.

Use Cases

  • Form Success
  • Edit Object Success
  • Delete Object Success

Toast

Toast messages provide the user information about medium-priority events that they may want to reference later.

Toast messages, when they aren't dismissed, live in the notifications panel until the user dismisses them or the notification expires.

Anatomy

  1. App Icon: Application that is sending the notification. Only use in multi-application environments.
  2. Status Icon: Informational or Error. (use Snackbar for confirmations)
  3. Title (max: 35 characters): Short Title Describing the Event
  4. Description (max: 156 characters): Short Description of the event
  5. Dismiss: An icon allowing the user to remove the toast from their view. Do not use "dismiss" as an action, clicking the dismiss button dismisses the toast.
  6. Time: Use "now" if the event is occurring currently. If the event happened in the past, use "[x] [time_period] ago"
  7. Actions [optional] (max: 19 characters each) (2 actions maximum): Provide the user actions to remediate the event when possible.

Usage Guidelines

  • Provide a location for the user to navigate to from the toast.
  • Use timestamps relative to the user (ex. "1 hour ago" instead of "May 11, 12:34pm")
  • Use a short, succinct summary message as the title. The title should be a short, consumable phrase that quickly communicates the high-level messaging in the toast. The maximum length for the title is 35 characters.
  • Use a consumable description for the toast message. The maximum length for the description is 156 characters.
  • Use "Dismiss" as an action. Clicking the dismiss button dismisses the toast.
  • Use more than 2 actions in a toast. If you need to provide more actions, surface the 2 most important actions, then allow the user to navigate to the object's detail view.
  • Use object identifier in the title. (ie. “VM12324235324234”). Use object identifiers in the description area as needed.
  • Add icons to the toast. The toast communicates high-level information quickly; adding iconography confuses the high-level meaning.
  • Show toasts when the condition or event is no longer relevant to the user. For example, if a subscription had expired, but was renewed by the user, the “Subscription Expiring Soon” notification is no longer relevant.

Use Cases

  • Configuration Change Initiated by Co-Admin, Security Implication (user added to organization)
  • Credit Available for Account
  • Requesting Approval from Administrator
  • Event that Impact User Account in Future Time (SDDC Maintenance Will Occur, Subscription Expiring in 30 Days)

See Notifications Usage for CSP's Snackbar Usage, and VMC Notification for VMC's.

Long Running Operations

When users perform an action that is asynchronous, the action triggers an API request to the backend. If the request is submitted successfully, the backend will begin the action. If the request fails to be submitted, the progress won't start.
Asynchronous tasks usually take longer to complete, consequently the user may want to move on to other tasks. The user may want to be informed of the long-running operation’s status in their current context, and may want to continue to other tasks upon the operation’s completion.

The Long-Running Operation Component allows the user visibility into a long running operation's status while providing them the opportunity to accomplish other tasks. Insure the Long Running Operation Panel persists throughout the views the user visits during while the operation is running. If a user's session ends, re-populate the Long Running Operation with its current status when the return. If the operation has completed or failed since the user's last session, inform them with the appropriate notifications.

States of the long running operation panel

1. Request submitted

When a user triggers an asynchronous task, the request can fail or succeed.

  1. If the request fails, show an alert banner on the page where the action was performed. The alert banner should be close to the action button.
  2. If the request is successfully submitted, send the user a notification using the long running task panel. Do not send this notification in a toast. Once the backend operation starts, transition the long running task panel into the in-progress state (see #2 below).

2. In progress

A progress state should be displayed as soon as backend operation begins.

2.1. Anatomy

  1. Title - Title is required for all operations. The title shouldn't exceed 35 characters to ensure it fits on the panel.
  2. Time - Time counter is optional. Display time when this information is available.
  3. Collapse / expand button - This button should be displayed on all operations.
  4. Progress bar - Progress bar is required for all operations. If the operation doesn't have determinate steps, display the indeterminate progress bar.
  5. Progress counter - Display progress counter only when this information is available. If you are using indeterminate progress bar, don't display the progress counter.
  6. Description - The description may describe the operation in detail, or be used to display status information about the progress.
  7. Dismiss - A dismiss button is required on all operations
  8. Cancel - If an operation can be cancelled, you can display the cancel button.

2.2 In progress panel variations

1. Informative (with all possible elements): When possible, provide the user with the time to completion, percentage to completion, and the operation status message.

2. Minimal (with only required elements): If certain information is not available, provide the user a granular sense of how long it will take the operation to complete.

There are other variables between informative and minimal. You can choose to use as many or as few optional elements as possible based on availability of information and your use cases.

3. Collapsed (when the user clicks on the collapse button): Display the title and progress bar. Also display progress counter if available.

There are occasionally cases where users are allowed to cancel an operation.

4. Error

If the operation fails, use the following variables to inform the user.

1. Informative variable: An informative state with all possible elements includes the title (required), description (optional), actions buttons (optional), and dismiss icon (required). Use "VIEW DETAIL" to take the user to the context of the operation when applicable. Use"TRY AGAIN" to allow the user to easily re-initiate the operation easily.

2. Minimal variable: the minimum requirement of the error state includes the title and the dismiss button. Only use this variable if no additional information is available.

5. Success

When the operation completes, inform the user of the result.

1. Informative variable: An informative state with all possible elements includes the title (required), description (optional), actions buttons (optional), and dismiss icon (required). Use "VIEW DETAIL" to take the user to the context of the operation when applicable.

2. Minimal variable: the minimum requirement of the success state includes the title and the dismiss button.

You can choose to use as many or as few optional elements as possible based on availability of information and your use cases.

6. Multiple

If the user has multiple long running operations in progress at the same time, combine the operations into a collapsed view to preserve screen space.

6.1 Mix In progress

1. Default. The default stack view includes the title (required), time (optional), progress bar (required), progress count (optional) and the collapse button (required). The second item in this view includes only the required elements.

2. Collapsed. The collapsed view includes only the text to indicate how many tasks are currently running.

3. Detail. The user can click on one of the operation and view more detail. The detail view includes the same elements as the in progress state (see #2.2 above), and can be any variation of the in progress state. Instead of the collapse button, this detail view includes a back button to go back to the default view (6.1.1)

6.2 Mix error

If one item in a progress stack fails, use the following interactions.

1. Default. Show the error title of the operation with a dismiss icon.

2. Detail. If you have additional detail to display for the failed operation, allow the user to click on the error and show the detail view. This detail view follow the same rules as the regular error state (See #4 above), except with a back button.

6.3 Mix success

If one of the stacked progress completed, use the following interactions.

1. Default. Show the success title of the operation with a dismiss icon.

2. Detail. If you have additional detail to display for the completed operation, allow the user to click on the success message and show the detail view. This detail view follow the same rules as the regular success state (See #5 above), except with a back button.

6.4 Mix success & error

It's also possible to have a mix of success & error message in a stacked view.

Usage Guidelines

  • Use when the user triggers an asynchronous operation.
  • Allow the user to "Cancel" the operation. If there is no ability to "Cancel," the use case may require a different presentation.
  • Allow the user to "Dismiss" the operation if a Notifications Panel exists. "Dismissing" the operation would remove it from the main view and place it into the Notifications Panel.
  • Provide a link to "Try Again" if the operation fails, when possible.
  • Provide a link to the object's detail view if the operation is successful, when possible.
  • Allow "Dismiss" if no Notifications Panel exists.
