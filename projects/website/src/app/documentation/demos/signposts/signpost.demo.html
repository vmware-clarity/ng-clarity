Signposts are displayed in a popover dialog box and contain contextual help or information.

Usage

Use a signpost when you want to show a small amount of contextual help without taking the user out of the current context. Use sparingly as a supplemental element and not as a primary method of adding detail. Consider a signpost in the following scenarios:

  • When the amount of content won't fit in a tooltip
  • When you want to include a header, an image, or text/image links in the content
  • When the information presented needs to stay in view until the user interacts with another element

Anatomy

Like a tooltip, a signpost has an arrow or tail that extends out to the trigger element. However, the dialog box area is larger to fit more content. Signposts are designed to show a relatively small amount of content to help the user, but they are large enough to include a title, images, text links, or image links. A vertical scrollbar (browser default) can be used if the content exceeds the maximum height of the dialog.

The signpost is paired with a trigger element, such as an icon or button link.

The allowable dimensions of a signpost are shown in the following table.

Behavior

Clicking the trigger opens the signpost. It remains visible until the user clicks the close icon or clicks anywhere outside of the dialog box to dismiss it. To keep the interface uncluttered, only one signpost is displayed at a time. When a dialog is visible, clicking an icon to open another one automatically dismisses the previous dialog box.

Trigger icon and dialog box positioning

The default position for the dialog box is 8px to the right of the trigger icon.

The default position for the dialog box is 8px from the end of the text.

In tables, the icons can be aligned in a column.

Recommendations

  • The popover should not be so large that it dominates the screen.
  • Make sure that the popover doesn't obstruct other important information on the screen.
  • With the exception of text or image links, don't insert any actionable components, such as buttons, in the dialog box.
  • Don't rely on signposts as a primary method of displaying additional information - i.e. as a way to save space on a page.
  • Use sparingly as a device to add immediate, relevant information.

Code & Examples

The signpost component uses the *clrIfOpen structural directive (detailed documentation coming soon) on the signpost content to indicate clearly that said content is only present in the DOM when open. We heavily recommend using it for many reasons: better performance, making the intent clear in your own templates, and following a more natural lifecycle for any directives or components inside of the signpost.

If for some reason this behavior was not the one you wanted, for instance if you want screenreaders to read the signpost inline as opposed to when the user activates the trigger, we also support the signpost content without a *clrIfOpen directive on it. It will be created eagerly when the signpost itself initializes, and will not be destroyed or recreated until the signpost itself is. Please make sure you know exactly why you are omitting this directive if you end up doing so.

Default Signpost

The default signpost is shown to the right of the trigger icon with the content centered vertically in the middle of the trigger. It opens with a click action. Once open its contents can be interacted with. The Signpost is closed when clicking on the icon or by clicking anywhere outside the clr-signpost-content element.

Default Signpost

Position: right-middle

Signpost positions

There are twelve signpost positions to help place the popover in an appropriate position when shown. You control the position and direction by declaring a position that orients your Signpost content as you see fit. On a clr-signpost-content component add the input [clrPosition]="'position'" where position is one of the following:

  • top-X: the arrow will point down and the popover will be above the trigger in the X position where X = left, middle or right
  • bottom-X: the arrow will point up and the popover will be below the trigger in the X position where X = left, middle or right
  • left-X: the arrow will point right and the popover will be left of the trigger in the X position where X = top, middle or bottom
  • right-X: the arrow will point left and the popover will be right of the trigger in the X position where X = top, middle or bottom

Custom Triggers

We provide a default trigger but you can add one yourself. The signpost component provides the clrSignpostTrigger attribute directive that can be added to the clr-signpost-content child. To control the position of your signpost, add [clrPosition]="position" so that it knows how the Signpost should point.

Open at Point (Context Menu)

Signposts can be opened at a specific screen coordinate using the openAtPoint() method. This is useful for displaying contextual information at the cursor position, for example on right-click. Use the [clrSignpostHideTrigger]="true" input to hide the default trigger icon when the signpost is opened programmatically.
