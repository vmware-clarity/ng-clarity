A theme is the collection of visual attributes that specify the look and feel of your application.

Theming in Clarity

Clarity is based on a system of design tokens and aliases. This allows many theming functions to be achieved with a simple switch of the token or re-assignment of an alias. The Clarity dark theme accomplishes a complete switch from light mode to dark using this method and is contained in about 50 lines of code. Simple themes can be created using this approach. For more complex themes where there are changes in sizes, spacing, and layout, as well as aspects like typeface and type scale, the complexity increases.

Simple themes

Simple themes are built much like the Clarity dark theme, by remapping aliases. With a large palette of global colors, a complete set of sizing increments, and clearly named attributes, such as --cds-global-border-radius, you can achieve a different look and feel relatively easily.

Complex themes

A complex theme is one in which a simple remapping of an alias is insufficient to achieve the desired results. This could be because brand colors are not contained in the global palette, or because a different typeface or type scale is needed, The process of creating such advanced themes requires careful attention to many details to ensure accessibility and that components behave as expected.

Theme attributes

Type

The benefit of switching the typeface is enormous. There is no more effective way to make a substantial change to the visual appeal of an application. Switching out the typeface is in itself a simple task, but weighing the ramifications is more complex. Even substantially similar typefaces will have differing horizontal spacing causing lines to wrap in different ways. This might lead to unexpected results especially within components. You should thoroughly test a typeface across the entire system for a refined and successful theme.

Simple

At the simple level, you continue to use the Clarity type classes with all their size and space attributes as-is. This will be the least intrusive, resulting in the smallest layout and spacing changes. However, you should still complete a full system audit to review the details of the change.

Complex

At a more complex level, you can also adjust the size and space attributes of one or more type classes. This will lead to many more areas where careful attention is needed to assess the design changes.

Color

Identifying colors by their usage is a best practice for assessing changes and their ramifications when you want to create a theme that uses different colors than are used in Clarity.

  • Status – colors such as action, success, warning, and danger
  • Container – colors that define the attributes of containers
  • Interaction – colors provided to enable all interactive states
  • Utility – decorative colors used on components such as badges and tags
  • Type colors – for headings and paragraphs including links—normal, clicked, and visited

Semantic color, known as status-color in Clarity, uses a multi-color system. There is a primary color with additional lighter and darker companion colors. These are selected from the scales in the Clarity Global Tokens. By examining the logic behind the selection and use of these colors in any Clarity theme, their relationship will be apparent. A common use is for the main color to define the solid border around a container, while the tint is used for the light background and the shade is used for the hover state. These will differ in light and dark themes.

To build a theme, at a minimum, you must define the mapping for all of these aliases for Clarity color tokens. For more information, see Color.

For accessibility, do not rely on color alone to distinguish meaning. Add text to convey the color differences or use visual cues such as underline or icons. For more information, see WCAG: Use of Color.

Sizes and space

Clarity uses a space token system for specifying the sizes within components, and a layout-space system for controlling the space between components. These systems offer powerful control, not only of individual components with regard to how they are sized, but to the way they fit together to create a complete layout. Changing any of the spacing tokens, while being a powerful way to globally adjust the way your application uses whitespace, is also very advanced and requires a great deal of attention to detail to avoid unintended consequences.

Corners and shadows

Adjusting border radius, shadow density, and distance can create a substantial change in visual appeal. These adjustments are less difficult than other adjustments since there is virtually no domino effect.

Advanced theming

Altering base colors

The Clarity color system uses aliases to semantically define the usage of colors in your application. Changing the color to which an alias points is a simple level of changing the appeal of your application. This is limited to the Clarity color set as defined in the global tokens file. If you wish to have other colors to choose from, the base values must be changed. These consist of a scale of values, each described by their base HUE in HSL:

Status

  • (Success) Green - 93
  • (Action) Blue - 198
  • (Secondary Action) Violet - 282
  • (Warning) Ochre (hue shift 41-35)
  • (Danger) Red - 9

Utility

  • Lavender - 238
  • Azure - 211
  • Aqua - 184
  • Jade - 160
  • Yellow (hue shift 40-50)
  • Tangerine - 25
  • Magenta - 345
  • Pink - 324

Muted

  • WarmGray - 282
  • CoolGray - 211
  • Slate - 238
  • Ice - 211
  • Tan - 41

Construction

  • Construction - 198

Changing this set will create a (color-only) theme, and if done with extreme care might not require the changing of any aliases. Beware that this set was carefully chosen to work with the aliases and it results in fully accessible components in every case. While it might be tempting to shift only the HUE and allow the scales of lightness and saturation to remain as-is, it is important to remember that different hues result in different luminance levels with the same lightness and saturation, and consequently will not have the same accessibility values as one another. To preserve accessibility, the scales will not always be able to retain the increments in the Clarity globals. Shifting the base color tokens will end up being a large project to ensure everything is working properly.

Recommendation: Switching themes as a testing method

The best way to create a Clarity theme is to begin with singular adjustments and use these to create a style sheet that can be previewed. By checking your work step-by-step, and running the changes through a browser accessibility check, you can ensure that both your aesthetic and accessibility needs are proceeding as desired. When the theme is behaving as desired, it is time to make other changes and build your application with your theme. Remember to test throughout the process. Accessibility is always a very important factor and when colors shift, they can cross the line from accessibility-compliant to non-compliant easily.

Themes

Clarity offers two default themes—light and dark. These themes can be used without modification to optimize user experience or they can be used as a guideline for creating custom themes in Clarity.

Light Theme

The light theme is Clarity’s default theme. It is best suited to content-rich applications, where users will be working with the application in well-lit conditions for brief periods of time throughout the day. A light theme can help prevent eye strain in applications where a user is expected to read a large amount of content — like documentation or a blog.

Dark Theme

The dark theme is best suited for applications that are content-sparse, yet highly interactive. A dark theme can help with eye strain if the user works with an application over a long span of time, but in a way that requires focus but not a great amount of reading.
