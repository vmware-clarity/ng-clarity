@clr/addons/wizard provides a multi-step wizard built on top of Clarity Modal. It adds workflow orchestration, automatic data injection between steps, loading indicators, standardised error handling, conditional step skipping, and an optional "Ready to Complete" summary page.

Wizard Structure

AppFx Wizard is built using Clarity Wizard and provides a lot of additional features. A wizard built using AppFx Wizard has the following structural elements:

  1. Wizard component — hosts <appfx-wizard>, configures steps, defines data retrieval per page, and handles Finish via CloseHandler.
  2. Wizard model — a POJO with Var<T> properties for initial state, page inputs/outputs, and data retrieval.
  3. Page component (per step) — implements StepModelHolder to receive the typed model, and optionally OnStepActivate / OnStepValidate.
  4. Page model (per step) — typed POJO with @In() and @Out() decorated properties.

Features

  • Pre-loads data for the first page; spinner displayed automatically
  • Orchestrates data retrieval on page transitions (lazy or eager)
  • Caches data between back/forth navigation
  • Reloads data when upstream selection changes
  • Optional summary page with step.summary
  • Custom footer component support
  • Conditional step relevance via step.isRelevant
  • Inline (Stepper) mode — same steps, no modal

Data Retrieval

Wizard automates the common data-loading concerns of a multi-step workflow so page components stay focused on rendering and stay agnostic of how their data is retrieved:

  • Pre-loads the data for the first page; a spinner is shown and user interaction is blocked until it resolves.
  • Orchestrates retrieval between steps — lazily (once the user clicks Next) or eagerly (as soon as an upstream selection changes).
  • Caches data so moving back and forth without changing a selection doesn't reload it.
  • Reloads data automatically when the user changes an upstream selection.
  • Transforms data before handing it to a page component — sub-setting, localizing, or mapping to a UI model.
  • Handles errors — if retrieval fails, a standardised error page replaces the active step, with a Retry action that resumes the workflow on success.

Ready to Complete Page

When at least one step defines step.summary, the wizard automatically appends a "Ready to Complete" page built on Property View. A section is generated for every step that defines summary, titled with step.navTitle || step.title.

Custom Wizard Validation

Long-running finish logic belongs in CloseHandler.onSubmit:

  • Throwing an Error displays it above the wizard pages and keeps the wizard open.
  • Returning an Observable<any> keeps the wizard open until it completes.
  • Throwing preventDisplayingWizardError keeps the wizard open without showing an error — the consumer is responsible for handling that error itself.

Custom Footer Component

Set footer.componentClass to render a component on the left side of the Back/Next/Finish buttons. The component must implement the WizardFooter interface (currentStep, steps, workflowModel), which the wizard re-injects on every step change so the footer can react to the current step.

Accessibility

Wizard is WCAG 2.1 compliant and supports 200% and 400% browser zoom when the application provides a ZoomLevelService. At 200% zoom the step navigation sidebar collapses behind a burger menu at the top-left of the dialog to free up space for page content. At 400% zoom the wizard expands into full-screen mode.

Debug Mode

Debug mode visualises the live state of the wizard model. Enable it programmatically by setting WorkflowConfigurationService.debug = true, or at runtime via appfx.debug=true in browser local storage. Once enabled, an info icon appears next to the wizard title; clicking it opens a signpost showing the current model properties and their values. Try it with the Debug Mode toggle in the Playground tab.

WizardComponent Inputs

Input Type Description
title string Wizard header title
pages Step[] Ordered step descriptors
wizardModel WorkflowModel Shared data model
opened boolean Two-way open/close binding
size string md / lg / xl / full-screen. Default: 'lg'
loading boolean Show the wizard's loading spinner
closeHandler CloseHandler onSubmit and onCancel callbacks (return Observable)
footer WizardFooterConfig Optional custom footer component class

WizardComponent Outputs

Output Type Description
openedChange EventEmitter<boolean> Emits when the opened input changes
onModelChange EventEmitter<ModelChange[]> Emits whenever a workflow model variable changes
onFinish EventEmitter<void> Emits when the user clicks Finish
onClose EventEmitter<Reason> Emits when the wizard closes via the close button, Cancel, or Finish. Use the Reason enum ('finish' | 'cancel') to distinguish the close origin.

Step Interface

Property Type Description
title string Step header title
description string? Subtitle shown below the title
navTitle string? Short title in left nav panel
componentClass Type<any> Angular component to render inside the step
model StepModel Typed step data model with @In/@Out properties
mappings Mappings? Custom property name remapping between step and workflow model
isRelevant Var<boolean>? Reactive relevance; step is skipped when false
summary (builder, model?) => SectionModel Builds this step's contribution to the summary page
recreateComponent boolean? Re-instantiate the component on every activation

Step Component Lifecycle Interfaces

Interface Method Description
StepModelHolder model: T Receives the typed step model before activation
OnStepActivate activate(changes) Called every time the step becomes active
OnStepValidate validate(): Observable<boolean> Called when user clicks Next; returning false keeps the step open

WizardFooter Interface

Implemented by the component referenced in footer.componentClass. The wizard re-injects these properties on every step change.

Property Type Description
currentStep Step The currently active step
steps Step[] All steps configured on the wizard
workflowModel WorkflowModel The shared wizard model

preventDisplayingWizardError

A sentinel Error exported by the addon. Throw it (or error the CloseHandler.onSubmit /onCancel Observable with it) to keep the wizard open without the default error banner — the consumer becomes responsible for surfacing that error itself.
